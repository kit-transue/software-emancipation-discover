November 12, 1997


Test Plan for External Languages Support

by

Scott Turner


External Language Support reimplements DISCOVER's information model construction. Here a strategy is presented to validate the new implementation.


Requirements


The goals of this project are first, to identify defects arising from the transition to ELS, and second to update the existing set of routine tests to work with the new version of DISCOVER that supports external languages. It is essential for defects to be sought thoroughly, since regressions in this core aspect of DISCOVER could undermine the product's usability in practically any way conceivable.


We need to catch problems in both modelbuild and interactive DISCOVER. But in addition to testing the complete system, it can be very useful to have tests that focus on the IF itself. Since we anticipate that most of the problems will be closely related to the IF, such problems can be understood and addressed more effectively if detected in the context of IF.


The kinds of failures to be detected include:

generation of incorrect IF syntax

generation of inconsistent IF content

generation of IF that does not correspond to the source code

generation of incomplete IF

catastrophic failure of the IF reader to process IF (correct or not)

failure of the IF parser to correctly handle correct IF

building an incorrect model


The essential tool for validating the new implementation is DISCOVER's existing test base. Unfortunately, much of the test base is dependent on data structures that will change in the new implementation. This impediment must be overcome.


Implementation


Overview


The work to be done has  four main aspects:

independent consistency testing of generated IF

validation of ELS support using batch/model tests

validation of ELS support using interactive/regression tests

establishment of routine use of tests revised for ELS

There are about 3000 interactive regression tests, 100 batch/model tests, and thousands of validation suite tests.


Independent Validation of Parser-generated IF


This will involve modifying the code for ELS support.


Checks will cover consistency of IF:

For each symbol, the kind is valid.

For each relation, the name is valid, and the source & target symbols have kinds consistent with the relation.

For each attribute, the name is valid, the value is consistent with the attribute name, and both are consistent with the kind of symbol which has the attribute.

If source files are accessible, then 'location' objects can be validated as referring to an actual position in the file.

Checks will cover completeness of IF:

For each variable, function, or constant symbol, check that it has a type relation.

This list of consistency and completeness checks is not yet complete. It should try to check everything that DISCOVER assumes.

Comparisons


The problematic aspect of testing the new implementation is to ensure that the new data structures are equivalent to the old. It is necessary to compare IF.


To compare the new data structures to the old:


1. Dump the astNode tree (dfa module) using IF syntax.


2. Compare 2 files of IF. This will parse the IF and produce a set of differences. It may dump the differences descriptively, something like the output of diff. Or it may dump the differences as the input to diff, so that its output may be substituted for the IF input, and the replaced files compared using diff as before.


3. The comparison function is non-trivial, since the IF is a graph. Symbols will be matched by name and kind within the global and class contexts. Relations will be matched based on their name, source and target. Then differences will be reported.

symbol absent from one file

symbol attributes or locations differ between files

relation absent from one file

relation attributes differ between files

AST tree or node difference


An alternative approach would make smaller modifications to the existing dumping, massaging it after it has been dumped, and then comparing it using diff. In my opinion there would be too many differences in form for external massaging to make this approach practical. 


Batch/Model Tests


The routine tests consist of

interactive regression tests (recorded)

batch tests

validation suites, e.g. Perennial

Normally the batch tests and validation suites do not dump the "AST" data structures. 


In these tests, ultimately what matters is that the new model matches the old. However, the value of comparing the IF is that these tests give a simpler test framework in which to validate that the new IF matches what the old product produced. Also, in case the models do not match, the IF comparison will provide a basis for tracking down the problem.


Test Details


The diagrams below show the model representation changes which occur in each kind of test. 


Batch:



The current batch test starts with source code and a saved dump. It constructs the model, saves it, restores it, and dumps. The new dump is compared to the saved.

Interactive:

The current interactive tests start with source code and a saved dump. It constructs the model and dumps. The new dump is compared to the saved.


Test of EDG parser:

A basic test of the EDG parser can be accomplished before we complete the ability to build the model from IF. This test starts with source code. It constructs an IF file using the EDG parser. It also runs the GCC parser, building a model, and dumping an IF file based on that model. The two IF files are compared.



Loopback:


A loopback test starts with just source code. It constructs the model using ELS techniques including IF. Then the model is dumped as IF. The two IF files are compared.


Other:

Note that the existing batch tests can still be run against an ELS-based DISCOVER. This could be a valuable testing mode, since it would not assume the ability to dump IF. The benefits of usnig IF are (1) it is easier to construct an intelligent comparison utility and (2) it can validate the EDG parser directly, for easier tracking down of problems.


Existing interactive tests would require more adaptation to be used in their current mode, because an ELS-based discover would not have any AST globals to dump. Still, running these tests would be valuable, because the importance of checking the AST globals is secondary. The testing would be modified to circumvent the comparison of AST globals.


Strategy


Once the comparison mechanism is implemented, using it to validate the ELS support might be rough sailing. Differences in ordering and in symbol numbering may make no substantive difference to the model, yet prevent the IF from being effectively compared using ‘diff’. To handle this difficulty, this plan anticipates making improvements comparing IF. Not only that, it allows the comparison to be tweaked from one test run to another by use of shell variables or something of the kind.


It may be particularly useful to have different levels of enforcement of matching one IF file to another. Then the checker can relax one aspect of enforcement and still validate other aspects as much as possible.


Where differences show up between the old AST and the new ELS AST, different resolutions are possible.

The new AST may be judged inappropriate for the IF, in which case the EDG-based parser would need to be modified.

The new AST may be considered an acceptable change, adding to the complexity of  IF-based validation of the ELS implementation.


The impact of superficial differences will not be limited to the comparison of the IF file to its corresponding astNode-based AST. Some differences can be expected in the SMT and DD as well. In cases of problematic noise, might they be dealt with using

modifications to SMT/DD dumping (as IF), or

modifications to diff in the form of a filter which precedes the application of diff

intelligent comparison of IF files based on the IF parser


Since there is likely to be a good deal of human analysis involved, the amount of effort to be put into this will be a judgment call.


Risks


First, despite the large number of existing tests, not every possible defect is covered.  But even within the scope of the existing tests, this plan is not perfect. The comparison of IF will not be perfect, nor will the generation of IF from AST nodes in the GCC-based product. These limitations of the plan are considered small; little is expected to “slip through the cracks”.


New Generation of Tests


The following ideas for future testing are preliminary, as there has been little discussion so far.


Some of the needed work deals with putting the existing tests in place for the ELS version of DISCOVER, beyond the transitional effort of ensuring that DISCOVER is functioning correctly and that we have the basis for a new generation of tests. In the future, the testing setup will need to be different from all of the methods diagrammed above. It will be analogous to the existing tests. Note that the analog of dumping AST globals will be to preserve the IF file used to build the model.


Will there be an awkward stage at which many of the tests are ready for automatic execution, while a substantial number are not yet validated due to large, impenetrable sets of diffs? In a way yes, but it need not be awkward. All that's needed is a list of tests that have not been validated, so that these can be run in transitional, validation mode. At this stage, the full set of tests will be run automatically against saved results that have been updated, recognizing that not every saved result is known to be correct.
